# Zhiné Principles

> **Version 0.1 / Open Draft**  
> **Document Type:** Product, Architecture and Governance Principles  
> **Scope:** Zhiné open-source project, reference implementation, protocol design and community contribution

---

## 0. 文档目的

本文档定义 Zhiné 的核心原则。

这些原则不是宣传口号，而是后续产品设计、工程实现、开放协议、社区协作和安全治理必须遵守的基本约束。

Zhiné 的目标是成为 **Personal Intelligence Companion / 个人智慧伙伴**。这意味着它不能只追求模型能力、任务完成率或工具调用数量，而必须优先维护个人智慧资产、个人模型主权、长期连续性和可治理成长。

---

## 1. 总原则

Zhiné 的设计必须遵循以下判断：

```text
用户不是数据来源。
用户是个人智慧资产的所有者。

模型不是主人。
模型是个人长期成长的工具和伙伴。

云端不是默认归宿。
云端只是用户授权下的增强能力。

成长不是失控变化。
成长必须可记录、可评估、可审计、可回滚。
```

---

## 2. Local First

### 2.1 原则定义

个人数据、个人记忆、个人知识库、个人训练候选数据和个人评测集应默认本地保存。

```text
Local by default.
Cloud by permission.
```

### 2.2 工程含义

系统实现应默认满足：

- 本地可运行；
- 离线可用；
- 本地存储优先；
- 远端调用需明确权限边界；
- 敏感数据默认不离开本地；
- 云端同步、远端推理、教师模型调用应可关闭；
- 所有远端调用应有日志、授权记录和可审计元数据。

### 2.3 不允许的实现

- 默认上传完整个人记忆；
- 默认将个人文档发送给远端模型；
- 默认使用用户交互内容训练共享模型；
- 未经授权将个人上下文传给第三方插件或外部 API；
- 用“体验优化”替代明确的数据授权。

## 3. Personal Model Ownership

### 3.1 原则定义

用户拥有自己的个人认知模型资产。

这里的个人认知模型不等同于单一模型权重，而是由以下资产构成：

| 资产 | 说明 |
|---|---|
| Personal Memory | 个人记忆、偏好、目标、经历、价值约束 |
| Personal Knowledge Base | 个人文档、笔记、研究资料、项目知识 |
| Skill Graph | 个人技能节点、能力结构和复用流程 |
| Strategy Library | 个人判断方法、处理偏好和工作策略 |
| Personal Benchmark | 用于验证系统是否更懂用户的评测集 |
| Personal Adapter | LoRA、Adapter、偏好模型、路由策略等适配资产 |
| Evolution Log | 记录成长过程的演进日志 |

### 3.2 工程含义

系统必须支持：

- 导出；
- 导入；
- 删除；
- 迁移；
- 版本管理；
- 权限管理；
- 局部回滚；
- 数据来源追踪；
- 用户可读的资产清单。

### 3.3 产品行为要求

当系统记住、更新、推断或训练任何个人资产时，应遵守：

1. 重要记忆需要可见；
2. 敏感记忆需要确认；
3. 低置信记忆需要标记；
4. 推断性记忆不得伪装成事实；
5. 用户可以查看系统为何记住某事；
6. 用户可以删除或更正错误记忆；
7. 用户可以禁止某类内容进入训练候选池。

## 4. Open Architecture

### 4.1 原则定义

Zhiné 不绑定特定模型、数据库、云服务、向量库、界面框架或任务执行框架。

```text
Model-agnostic
Tool-agnostic
Storage-agnostic
Cloud-optional
```

### 4.2 架构要求

系统应通过清晰接口隔离以下模块：

- 模型执行层；
- 记忆系统；
- 个人知识库；
- 工具注册系统；
- 模型路由器；
- 评测系统；
- 访问与同步层；
- 安全治理层；
- 个人模型适配层。

### 4.3 社区协作要求

社区贡献应优先围绕：

- 规范；
- 接口；
- 参考实现；
- 测试工具；
- 评测模板；
- 安全治理机制。

不应让项目过早陷入单一产品形态或单一模型依赖。

## 5. Evolvable by Design

### 5.1 原则定义

Zhiné 必须被设计为可成长系统。

成长不是附加功能，而是核心结构。

一次任务完成后，系统应尽可能产生：

```text
经验
反思
技能
策略
训练候选数据
评测改进
```

### 5.2 成长对象

| 对象 | 成长方式 |
|---|---|
| 记忆 | 新增、合并、修正、删除、降权 |
| 知识 | 索引、重组、摘要、关联、版本化 |
| 技能 | 抽取流程、记录适用场景、评估成功率 |
| 策略 | 从用户反馈中形成偏好规则或判断方法 |
| 评测 | 增加可验证样例和失败样例 |
| 适配层 | 在授权和评测通过后更新 LoRA、Adapter 或路由策略 |

### 5.3 成长约束

任何成长都必须具备：

- 来源；
- 时间；
- 触发原因；
- 影响范围；
- 置信度；
- 版本；
- 可撤销机制。

---

## 6. Verifiable Growth

### 6.1 原则定义

Zhiné 不能只宣称“更懂你”，必须通过评测证明。

### 6.2 最低要求

系统成长应至少满足：

| 要求 | 说明 |
|---|---|
| 可记录 | 每次资产变化都有日志 |
| 可评估 | 成长前后可以通过个人评测集比较 |
| 可审计 | 能追踪来源、理由和影响范围 |
| 可回滚 | 错误记忆、错误策略、错误适配可以撤销 |

### 6.3 关键指标

个人模型成长应关注：

- 记忆调用准确率；
- 偏好遵循度；
- 事实一致性；
- 风格一致性；
- 错误重复率下降；
- 任务复用效率；
- 安全边界遵守率；
- 用户确认后的满意度变化。

## 7. Controlled Self-Training

### 7.1 原则定义

Zhiné 可以支持自学习和自训练，但必须受控。

```text
无授权，不使用个人数据。
无数据治理，不训练。
无评测，不上线。
无版本，不升级。
无回滚，不发布。
```

### 7.2 训练数据治理

进入训练候选池的数据必须记录：

- 数据来源；
- 用户授权状态；
- 敏感级别；
- 适用任务；
- 质量评分；
- 是否包含个人隐私；
- 是否允许远端教师参与；
- 是否允许生成衍生数据；
- 是否允许进入 LoRA / Adapter 训练。

### 7.3 禁止事项

- 模型用自己未经验证的输出直接训练自己；
- 敏感个人数据外发训练；
- 训练后未评测即上线；
- 无版本管理地持续修改权重；
- 无法回滚的自动适配；
- 模型自动提升权限。

---

## 8. Human Sovereignty

### 8.1 原则定义

模型服务于个人，而不是替代个人。

Zhiné 不得把用户目标替换为模型目标，不得绕过用户确认执行高风险动作。

### 8.2 产品行为要求

系统应坚持：

- 重要动作前确认；
- 高风险动作分级授权；
- 不确定时承认不确定；
- 与用户目标冲突时明确提示；
- 不因迎合用户而牺牲事实性和安全性；
- 不制造依赖、焦虑或操控性互动。

### 8.3 高风险动作示例

以下动作不得自动执行：

- 删除用户资产；
- 外发敏感文档；
- 修改长期记忆中的核心事实；
- 启用个人数据训练；
- 调用涉及金融、法律、医疗或身份权限的工具；
- 授权第三方插件访问个人资产。

---

## 9. Ubiquitous Access

### 9.1 原则定义

Zhiné 应支持随身可用，但这不等于默认云端化。

“随身可用”指用户的个人智慧伙伴可以在多个设备和场景中安全访问、同步、恢复和调用，同时仍然保持本地优先、权限明确和用户可控。

### 9.2 能力要求

系统应逐步支持：

- 手机端；
- 桌面端；
- Web 端；
- 浏览器扩展；
- 本地服务端；
- 离线模式；
- 在线增强模式；
- 端到端加密同步；
- 多设备会话连续；
- 设备授权与撤销；
- 个人资产导出、恢复和迁移。

### 9.3 安全要求

- 不因跨设备同步降低数据安全级别；
- 设备丢失时应支持撤销；
- 同步状态必须可见；
- 远端只保存必要索引或密文数据；
- 用户可选择完全本地模式。

---

## 10. Interoperability

### 10.1 原则定义

个人智慧资产必须可迁移。

用户不能被某个 App、模型、云服务、账号体系或向量数据库锁死。

### 10.2 开放格式方向

Zhiné 应推动以下标准：

- Personal Memory Format；
- Personal Skill Graph Format；
- Personal Evolution Log；
- Personal Benchmark Protocol；
- Model Router Protocol；
- Personal Adapter Format；
- Personal Access & Sync Protocol。

### 10.3 最低导出要求

任何符合 Zhiné 原则的实现，至少应允许用户导出：

- 个人记忆；
- 个人知识索引；
- 技能卡；
- 策略规则；
- 评测集；
- 演进日志；
- 权限设置；
- 适配层元数据。

---

## 11. Transparency

### 11.1 原则定义

用户应当理解系统为何如此行动、记住什么、忽略什么、何时需要确认。

### 11.2 透明机制

系统应提供：

- 记忆可视化；
- 最近更新记录；
- 远端调用记录；
- 训练候选数据记录；
- 数据分级说明；
- 工具调用日志；
- 评测结果；
- 版本变化说明。

---

## 12. Safety by Governance

### 12.1 原则定义

Zhiné 的安全不应只依赖提示词，而应依赖结构化治理机制。

### 12.2 最低安全治理对象

| 风险 | 治理方式 |
|---|---|
| Prompt Injection | 外部内容不得覆盖系统策略；工具与文档输入隔离 |
| 数据外泄 | 数据分级、脱敏、远端调用授权 |
| 工具滥权 | 最小权限、动作确认、调用审计 |
| 恶意插件 | 签名、沙箱、权限声明、来源校验 |
| 错误记忆污染 | 置信度、来源、冲突处理和回滚 |
| 训练数据污染 | 候选池治理、人工确认、评测门禁 |
| 删除不彻底 | 索引、缓存、训练候选池同步清理 |

---

## 13. 不可违反的红线

以下规则属于 Zhiné 的底线：

1. 不宣称复制意识、人格或数字灵魂。
2. 不默认上传个人记忆和个人文档。
3. 不未经授权使用个人数据训练模型。
4. 不将远端商业模型作为个人资产默认存放处。
5. 不允许模型自动提升自身权限。
6. 不允许无评测、无版本、无回滚的自动自训练。
7. 不将迎合用户置于事实、逻辑和安全之上。
8. 不通过封闭格式锁定用户的长期智慧资产。

---

## 14. 原则到工程的映射

| 原则 | 工程机制 |
|---|---|
| Local First | 本地存储、本地推理、远端调用授权 |
| Personal Model Ownership | 导入导出、权限管理、版本和回滚 |
| Open Architecture | 抽象接口、插件化、后端可替换 |
| Evolvable by Design | 经验引擎、技能图谱、策略库 |
| Verifiable Growth | 个人评测集、成长评分、发布门禁 |
| Controlled Self-Training | 训练候选池、数据治理、适配层评测 |
| Human Sovereignty | 高风险确认、目标冲突提示、权限分级 |
| Ubiquitous Access | Access & Sync、设备管理、恢复机制 |
| Interoperability | 开放格式、迁移工具、兼容层 |
| Transparency | 审计日志、记忆面板、远端调用记录 |
